# ☀️ Morning Briefing: AI 자동 뉴스 브리핑 시스템

> **개발 기록 및 아카이브 문서**
> 기존 Make.com 기반의 단순 자동화를 넘어, 완벽하게 커스터마이징 가능한 나만의 AI 뉴스 큐레이션 및 음성 브리핑 플랫폼으로 진화한 과정을 기록합니다.

---

## 1. 프로젝트 개요 (Project Overview)
**Morning Briefing**은 매일 아침 사용자가 설정한 관심 분야의 글로벌 뉴스를 수집하고, 최신 AI 모델(GPT-4o)을 활용해 아나운서 톤의 대본으로 요약한 뒤, 인간과 구분이 어려운 고품질 AI 음성(TTS)으로 합성하여 이메일(또는 대시보드)로 배달해 주는 **개인화된 AI 비서 서비스**입니다.

## 2. 개발 배경 및 목적 (Background & Objectives)
* **초기 상태**: Make.com 등의 노코드 툴에 의존하여 유연한 기능 확장(예: 다중 사용자 지원, 세밀한 프롬프트 튜닝, 자체 데이터베이스 구축)에 한계가 있었습니다.
* **목표**:
  1. 외부 노코드 툴에 의존하지 않는 100% 독립적인 자체 Node.js 백엔드 구축
  2. 향후 서비스 확장을 고려한 **다중 사용자(Multi-User)** 지원 및 개인별 설정(관심사, 목소리 톤, 시간대) 기능 구현
  3. 다른 프로젝트들과 하나의 Supabase 데이터베이스를 공유하면서도 완벽하게 독립적으로 동작하는 아키텍처 설계

## 3. 기술 스택 (Tech Stack)
* **Backend**: Node.js (커스텀 라우팅 시스템 도입으로 가볍고 빠른 API 서버 구축)
* **Frontend**: Vanilla JavaScript, HTML5, CSS3 (프레임워크 없이 빠르고 직관적인 UI 구현)
* **Database & Auth**: Supabase (PostgreSQL) + Row Level Security(RLS)
* **AI & 3rd Party APIs**:
  * **OpenAI API**: 대본 작성(GPT-4o-mini) 및 고품질 음성 합성(OpenAI TTS)
  * **NewsAPI**: 전 세계 주요 뉴스 데이터 수집
  * **Nodemailer**: SMTP를 활용한 자동 이메일 발송

## 4. 핵심 기능 (Core Features)
1. **자동화 파이프라인 엔진 (`runner.js`)**: `NewsAPI 수집 ➡️ LLM 대본 작성 ➡️ TTS 음성 합성 ➡️ Supabase Storage 업로드 ➡️ 이메일 전송 ➡️ DB 로그 기록`으로 이어지는 물흐르는 듯한 자동화 파이프라인.
2. **독립된 DB 네임스페이스 (`a_` 접두사)**: 단일 Supabase 프로젝트 내에서 5개 이상의 개별 프로젝트가 돌아가는 환경을 고려해, 본 프로젝트의 모든 테이블과 트리거는 `a_` 접두사를 사용하여 충돌을 원천 차단했습니다.
3. **다중 사용자 맞춤 설정**: `a_user_settings` 테이블을 통해 사용자마다 원하는 뉴스 카테고리, LLM 모델, TTS 목소리, 재생 속도 등을 다르게 설정할 수 있습니다.
4. **동시성 제어 및 캐싱 (Phase 3)**: 매 정시에 여러 명의 스케줄이 동시에 실행될 때 API Rate Limit을 초과하지 않도록 세마포어(Semaphore) 패턴의 병렬 처리기를 구현하고, 동일한 뉴스 요청에 대한 인메모리 캐싱을 도입했습니다.

## 5. 단계별 개발 여정 (Development Journey)

### 📍 Phase 1: MVP 기반 다지기
* 기존 노코드 로직을 Node.js 환경으로 완벽히 마이그레이션.
* `news.js`, `llm.js`, `tts.js`, `storage.js` 모듈 분리.
* 로컬 환경에서 CLI 명령어로 단일 브리핑 파이프라인이 끝까지 정상 동작하는 것을 확인.

### 📍 Phase 2: 다중 사용자(Multi-user) 및 대시보드 구축
* 단순 스크립트 실행 방식을 넘어 웹 대시보드(Frontend)와 API 서버(Backend) 구조로 발전.
* Supabase Auth 연동을 통한 로그인 구현.
* 단일 파일이었던 `dashboard.js`를 정적 파일 서빙 방식(`app.js`, `index.html`)과 API 라우터(`router.js`) 구조로 모듈화하여 확장성 확보.

### 📍 Phase 3: 고도화 및 최적화
* 여러 사용자가 동시에 브리핑을 생성할 때 뻗지 않도록 `MAX_CONCURRENCY` 동시성 제어 도입.
* 같은 국가/카테고리를 구독하는 사용자들이 동시에 실행될 때 NewsAPI를 한 번만 호출하도록 `10분 TTL 인메모리 캐시` 구현.

### 📍 Phase 4: 통합 및 트러블슈팅 (The Final Sprint)
* **DB 트리거 충돌 해결**: 기존 공용 트리거 이름(`on_auth_user_created`)이 다른 프로젝트와 충돌하는 문제를 발견하고 `a_on_auth_user_created` 로 안전하게 마이그레이션.
* **유니크 인덱스 버그 픽스**: Phase 1 시절의 1인용 설계(하루 전체 1개 브리핑만 허용)로 인해 멀티유저 환경에서 DB insert가 거부당하는 이슈를 발견. `a_briefing_logs_user_date_unique` (유저별 하루 1개) 인덱스로 수정하여 완벽히 해결.
* **이메일 배달 시스템 점검**: Gmail SMTP와 앱 비밀번호를 연동하여 스팸 이슈 등 실제 환경에서의 배달 사고를 추적 및 복구.

## 6. 최종 운영 아키텍처 및 향후 발전 (Zero-cost Deployment & Roadmap)

이 프로젝트는 서버 유지비용이 전혀 발생하지 않는 **"무과금 듀얼 배포 아키텍처"**로 설계 및 운영됩니다.

1. **백엔드 자동화 (GitHub Actions)**
   * 코드를 24시간 깨어있는 서버(Railway/AWS)에 올리면 월 유지비용이 발생합니다. 대신, GitHub Actions의 Cron 스케줄러 기능을 활용해 매 정시마다 잠시 깨어나 파이프라인(`npm run pipeline`)만 실행하고 종료되도록 구성했습니다. 이를 통해 **서버 비용 0원**으로 평생 운영이 가능합니다.
2. **프론트엔드 대시보드 (Vercel)**
   * 사용자가 설정을 변경하거나 히스토리를 확인할 때 사용하는 정적 웹/API 라우터는 Vercel에 배포하여 빠른 속도와 무료 호스팅의 이점을 극대화합니다.

**🚀 향후 고도화 모듈**
* **다양한 정보 소스 연동**: 암호화폐 동향, 개인 관심 블로그 RSS, 날씨 정보 등을 모듈로 추가하여 LLM 프롬프트에 주입.
* **알림 채널 다각화**: Telegram Bot API나 Slack Webhook을 연동하여 모바일 메신저로 즉각적인 Push 알림 지원.
* **대시보드 UI 고도화**: 커스텀 프롬프트 입력창, 현지 타임존 설정 등 고급 기능을 프론트엔드에 추가 연동.
